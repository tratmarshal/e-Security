# Backend Architecture

เอกสารนี้บันทึกแนวทาง backend ของระบบ Warrant Tracker หลังจากปรับโครงสร้างใหม่ ใช้คู่กับ `CONTEXT.md` โดย `CONTEXT.md` เก็บภาษากลางของโดเมน ส่วนเอกสารนี้เก็บขอบเขตระบบและบทบาทไฟล์

## Data Stores

ระบบใช้ Google Spreadsheet แยกกัน 2 ไฟล์

1. `ฐานข้อมูลหมายจับ`
   - เก็บข้อมูลหมายจับตั้งแต่ปี 55 ถึงปัจจุบัน
   - เป็นแหล่งข้อมูลหลักของหมายจับและจำเลย
   - ใช้ค่าหลักของฐานข้อมูลหมายจับเสมอ เช่น `สิ้นผลรอเพิกถอน`

2. `รายการอัปเดต`
   - เก็บรายการแก้ไขที่ front end ส่งเข้ามาในแท็บ `processing`
   - เก็บประวัติการใช้งานของเจ้าหน้าที่ในแท็บ `log`
   - `processing` เป็นพื้นที่พักรายการแก้ไข ไม่ใช่ฐานข้อมูลหมายจับตัวจริง

## Processing Flow

1. Front end ส่งรายการแก้ไขเข้า backend
2. Backend บันทึกรายการแก้ไขลงแท็บ `processing`
3. `auto.gs` อัปเดต `ฐานข้อมูลหมายจับ` ตามรายการแก้ไขล่าสุดแบบเรียลไทม์
4. เมื่อมีหลายรายการแก้ไขของหมายจับเดียวกัน ให้ใช้ `ลำดับรายการแก้ไข` เป็นตัวตัดสินรายการล่าสุด
5. Front end อ่านรายการจาก `processing` กลับไปแสดงผลได้
6. Front end แสดงสถานะการสะท้อนข้อมูลเป็นจุดสีเท่านั้น

## Status Display

`processing` สามารถใช้คำแสดงผลที่เหมาะกับหน้าทำงาน เช่น `รอเพิกถอน`

ก่อนเขียนลง `ฐานข้อมูลหมายจับ` backend ต้องแปลงเป็นค่าหลัก เช่น

| พื้นที่ทำงาน / processing | ฐานข้อมูลหมายจับ |
| --- | --- |
| `รอเพิกถอน` | `สิ้นผลรอเพิกถอน` |

แนวทางนี้ช่วยให้ฐานข้อมูลหลักใช้ภาษาทางการและมั่นคง ส่วนหน้าทำงานยังใช้คำสั้นที่อ่านง่ายได้

## Internal Sync Status

แท็บ `processing` เก็บค่าสถานะภายในเพื่อให้ระบบทำงานเร็วและตรวจปัญหาได้ง่าย

| ค่า | ความหมาย | การแสดงผลบน front end |
| --- | --- | --- |
| `pending` | ยังไม่สะท้อนลงฐานข้อมูลหมายจับ | จุดแดง |
| `synced` | สะท้อนลงฐานข้อมูลหมายจับแล้ว | จุดเขียว |
| `error` | สะท้อนไม่สำเร็จ | จุดแดง |

Front end ไม่ต้องแสดงข้อความ `pending`, `synced`, หรือ `error` ให้ผู้ใช้ทั่วไปเห็น

## Backend Modules

### Front-end Service Layer

ไฟล์กลุ่มนี้เป็นทางเข้าหลักที่ front end ใช้สื่อสารกับ backend

- `auth.gs`: ตรวจสอบสิทธิ์และยืนยันตัวตนผู้ใช้
- `search.gs`: คำสั่งค้นหาหมายจับและข้อมูลที่เกี่ยวข้อง
- `process.gs`: คำสั่งสร้างหรือปรับรายการดำเนินการและรายการแก้ไข
- `user.gs`: จัดการข้อมูลผู้ใช้

### Internal Shared Layer

ไฟล์กลุ่มนี้เป็นของใช้ร่วมภายใน backend ไม่ควรเป็น API ให้ front end เรียกโดยตรง

- `var.gs`: ประกาศค่าคงที่และตัวแปรกลาง
- `config.gs`: ตั้งค่าระบบที่ควรแยกจากค่าคงที่ทั่วไป
- `utils.gs`: ฟังก์ชันช่วยเหลือทั่วไป
- `cache.gs`: ชั้นอ่านข้อมูลเร็วของ backend เพื่อลดการอ่านชีทซ้ำ

### Audit Layer

- `log.gs`: บันทึกประวัติการใช้งานของเจ้าหน้าที่ เช่น ค้นหา แก้ไข กดดำเนินการ เข้าจากหน้าใด และเมื่อใด
- `log.gs` ถูกเรียกจากคำสั่ง backend ที่ต้องบันทึกร่องรอยการใช้งาน
- ประวัติการใช้งานแยกจากรายการดำเนินการและรายการแก้ไข

### Sheet Automation Layer

- `auto.gs`: อัปเดตฐานข้อมูลหมายจับตามรายการแก้ไขล่าสุดจาก `processing`
- `auto.gs` เป็นงานเบื้องหลังหรือ trigger ไม่ใช่ API ที่ front end เรียกโดยตรง
- หลังอัปเดตฐานข้อมูลหมายจับแล้ว ควรปรับสถานะภายในของรายการแก้ไข และล้างหรือรีเฟรช cache ที่เกี่ยวข้อง

## Cache Policy

`cache.gs` เป็น helper ภายใน backend สำหรับลดการโหลดข้อมูลจาก Spreadsheet ซ้ำ

แนวทางการอ่านข้อมูล:

1. service เช่น `search.gs` หรือ `process.gs` ขอข้อมูลจาก `cache.gs`
2. ถ้า cache ยังสด ให้ใช้ข้อมูลจาก cache
3. ถ้า cache ไม่มีหรือหมดอายุ ให้อ่านจาก Spreadsheet แล้วอัปเดต cache
4. เมื่อ `auto.gs` เปลี่ยนข้อมูลหมายจับ ต้องล้างหรือรีเฟรช cache ที่เกี่ยวข้อง

Front end ไม่ควรรู้ว่าข้อมูลมาจาก cache หรือ Spreadsheet

## Migration Plan

การปรับโครงสร้างจากโค้ดเดิมควรทำแบบค่อยเป็นค่อยไป เพื่อลดโอกาสกระทบระบบที่ใช้งานได้อยู่

1. เปลี่ยนบทบาท `sync.gs` เป็น `auto.gs`
   - ใช้สำหรับอัปเดต `ฐานข้อมูลหมายจับ` จากรายการแก้ไขใน `processing`
   - คงแนวคิดสถานะภายใน `pending`, `synced`, และ `error`
   - เพิ่มการแปลงค่าจาก `รอเพิกถอน` เป็น `สิ้นผลรอเพิกถอน` ก่อนเขียนฐานข้อมูลหมายจับ

2. แยก `db.gs` ตามหน้าที่จริง
   - ย้าย `searchWarrant` ไป `search.gs`
   - ย้าย `addProcessRecord`, `getPendingProcess`, และ `markProcessRevoked` ไป `process.gs`
   - ย้าย `logActivity` ไป `log.gs`
   - ย้าย helper กลาง เช่น lock, เวลา, การ normalize ข้อความ, และ helper จัดการชีท ไป `utils.gs` หรือไฟล์ช่วยภายในที่เหมาะสม

3. ปรับ `var.gs` ให้ตรงกับภาษากลาง
   - สถานะหมายจับใช้ `ต้องการตัว`, `สิ้นผลรอเพิกถอน`, และ `เพิกถอน`
   - สถานะสำนวนใช้ `รอดำเนินการ`, `เสนอศาล`, `รอส่งต่อ`, และ `ส่งต่อแล้ว`
   - ค่าที่ใช้แสดงใน `processing` เช่น `รอเพิกถอน` ต้องไม่ปนกับค่าหลักของฐานข้อมูลหมายจับ

4. เพิ่ม `cache.gs`
   - ใช้เป็นชั้นอ่านข้อมูลเร็วภายใน backend
   - ให้ `search.gs` และ `process.gs` อ่านผ่าน cache เมื่อเหมาะสม
   - ให้ `auto.gs` ล้างหรือรีเฟรช cache หลังเปลี่ยนฐานข้อมูลหมายจับ

5. ปรับ front end หลัง backend นิ่งแล้ว
   - ให้ front end เรียก service layer ที่แยกใหม่
   - แสดงสถานะ `pending`, `synced`, และ `error` เป็นจุดแดงหรือเขียวเท่านั้น
   - ไม่ให้ front end เรียก `var.gs`, `config.gs`, `utils.gs`, `cache.gs`, หรือ `auto.gs` โดยตรง

## Warrant and Case Flow

ระบบแยกสถานะหมายจับและสถานะสำนวนออกจากกัน

| ขั้น | เหตุการณ์ | สถานะสำนวน | สถานะหมายจับ |
| --- | --- | --- | --- |
| 1 | ศาลออกหมายจับ | `รอดำเนินการ` | `ต้องการตัว` |
| 2 | ได้ตัวหรือพบเหตุที่ทำให้หมายจับสิ้นผล | `เสนอศาล` | `สิ้นผลรอเพิกถอน` |
| 3 | ศาลพิจารณาแล้วสำนวนกลับมาตำรวจศาล | `รอส่งต่อ` | `สิ้นผลรอเพิกถอน` |
| 4 | ตำรวจศาลบันทึกเพิกถอน | `รอส่งต่อ` | `เพิกถอน` |
| 5 | ส่งต่อสำนวนหลังเพิกถอน | `ส่งต่อแล้ว` | `เพิกถอน` |

`รายการที่ดำเนินอยู่` คือรายการที่ยังไม่ถึงขั้น `ส่งต่อแล้ว`

`ประวัติรายการดำเนินการ` คือรายการที่ `สถานะสำนวน` เป็น `ส่งต่อแล้ว` และ `สถานะหมายจับ` เป็น `เพิกถอน`
